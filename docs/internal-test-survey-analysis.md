# 内测问卷反馈与项目实现分析

生成日期：2026-06-29

数据来源：`370656217_按序号_内测问卷收集_18_18.xlsx`，共 18 份反馈，提交时间集中在 2026-06-28 至 2026-06-29。本文只归纳问题，不记录问卷中的 QQ 号、IP、昵称等个人信息。

## 1. 结论摘要

内测反馈集中在 6 条主线：

| 类别 | 反馈热度 | 主要表现 | 对应模块 |
| --- | ---: | --- | --- |
| 注册与验证码 | 高 | QQ 邮箱验证码找不到、注册入口/验证码/密码概念混淆、提示字小 | Auth、Bot、Email、注册页 |
| 照片上传与照片冷却 | 高 | 5MB 限制、不能连续选多张、被驳回后 7 天才能再传 | PhotoUploader、照片 API、评分流程 |
| 资料/评分滑杆难用 | 中 | 身高、体重、年龄范围、评分滑杆在手机上费力 | 资料页、编辑页、评分页、DualRangeSlider |
| 单向匹配信息过少 | 中 | 只能看到“匹不匹配”，看不到兴趣、游戏、简介等 | `/api/matches`、单向匹配页、查看申请 |
| 匹配维度不足 | 中 | 希望增加 BMI、同省/地理细分、兴趣爱好、未来规划、思想价值观、身材分 | Profile、Preference、Matching |
| 浏览器兼容与可见状态 | 中 | 评分页需换浏览器、夸克浏览器选择题不显示已选 | CSS、表单控件、评分页、移动端适配 |

当前项目架构已经覆盖大部分核心业务对象：用户、资料、偏好、照片、评分、匹配快照、查看申请、举报、通知、审计。但有几处实现和问卷反馈高度吻合，需要优先修：

1. 举报入口把内部 `userId` 当成 QQ 号预填，导致用户看到非 QQ 内容，还要手动复制 QQ 才能举报。
2. 照片上传组件只读取 `files?.[0]`，文件输入也未启用 `multiple`，所以不能连续多选。
3. 匹配算法没有使用 Prisma 中已存在的 `locationScope / expectedProvinceCode / expectedCityCode`，实际只按年龄、身高、体重、属性筛选。
4. 单向匹配接口刻意只返回脱敏匹配项，不返回昵称/简介等轻量资料，导致信息密度低。
5. 评分任务 API 向评分端返回被评分人的出生日期、身高、体重、属性等资料，和项目文档“评分组只能看到照片”的隐私原则冲突。

## 2. 项目架构与反馈映射

项目是 Next.js App Router + React + TypeScript 全栈应用，后端使用 Prisma/PostgreSQL，Redis 处理验证码与队列状态，MinIO/S3 存储照片，NapCat Bot 负责 QQ 群注册指令与群成员信息同步。

关键模块：

| 模块 | 关键文件 | 和问卷反馈的关系 |
| --- | --- | --- |
| 注册/验证码 | `src/server/bot/register-command.handler.ts`、`src/lib/verification.ts`、`src/lib/email.ts`、`src/app/(auth)/signup/page.tsx` | 用户对“群内发指令、QQ 邮箱验证码、网页登录设置密码”的流程仍有理解成本 |
| 登录入口 | `src/app/(auth)/login/page.tsx` | 有用户把邮箱验证码输到密码栏，说明登录/注册入口切换和输入说明需要更强区分 |
| 资料创建/编辑 | `src/app/(auth)/complete-profile/page.tsx`、`src/app/(main)/profile/edit/page.tsx`、`src/app/api/profile/me/route.ts` | 7 天冷却、资料字段、滑杆体验、草稿/发布逻辑均在这里 |
| 照片上传 | `src/components/profile/photo-uploader.tsx`、`src/app/api/profile/photos/route.ts` | 5MB、单张上传、多图选择、上传须知、照片评分入口 |
| 匹配 | `src/lib/matching.ts`、`src/app/api/matches/route.ts`、`src/app/(main)/matches/*` | 单向/双向匹配、同省筛选、匹配维度、资料展示密度 |
| 查看申请 | `src/app/api/view-requests/route.ts`、`src/app/(main)/requests/page.tsx` | 单向匹配资料过少时，当前主要靠查看申请解锁 |
| 举报 | `src/app/(main)/report/page.tsx`、`src/app/api/reports/route.ts` | 问卷明确反馈举报弹窗/页面 QQ 号异常 |
| 评分 | `src/app/(scorer)/scoring/page.tsx`、`src/app/api/scoring/tasks/route.ts` | 评分页可见性、滑杆体验、评分端隐私 |

## 3. 需修复的 bug 与错误

### P0：举报入口传参错误

现象：用户从匹配页点举报时，举报页面预填的不是 QQ 号，而是类似内部 ID 的字符串。问卷反馈为“弹出的界面带出的 QQ 账号是乱码，还需要复制对面 QQ 号进行更改才能举报”。

代码证据：

- `src/app/(main)/matches/mutual/page.tsx:352` 使用 `/report?target=${match.userId}`。
- `src/app/(main)/matches/one-way/page.tsx:450` 同样使用 `/report?target=${match.userId}`。
- `src/app/(main)/report/page.tsx:151` 将该值作为 `targetUserId` 字段提交。
- `src/app/api/reports/route.ts:25` 和 `src/app/api/reports/route.ts:67` 实际把它当 QQ 号查询。

建议修复：

- 举报 API 支持两个明确字段：`targetUserId` 和 `targetQQ`，不要用 `targetUserId` 承载 QQ 号。
- 从匹配页进入举报时传内部 `targetUserId`，举报页展示“已选择举报对象”，不要求用户看到或复制 QQ。
- 后端校验目标用户是否存在，且最好校验举报人和目标存在匹配、查看申请或其他合理上下文，防止任意枚举用户 ID 举报。
- 如果仍允许手填 QQ，表单字段命名改为 `targetQQ`。

### P1：照片上传不支持连续多选

现象：用户反馈“传照片，不能连续选，每张选完就弹回去了”。代码中上传组件只取第一张文件，隐藏 input 也没有 `multiple`。

代码证据：

- `src/components/profile/photo-uploader.tsx:66` 只读取 `e.target.files?.[0]`。
- `src/components/profile/photo-uploader.tsx:250-253` 文件 input 只有 `type="file"` 和 `accept`，没有 `multiple`。
- `src/components/profile/photo-uploader.tsx:101` 每次只把一个 `newPhoto` 追加进状态。

建议修复：

- input 增加 `multiple`。
- `handleFileSelect` 改为读取 `Array.from(e.target.files ?? [])`，按剩余可上传数量截断。
- 逐张上传并展示进度，部分失败时保留成功项并列出失败原因。
- 使用函数式状态更新，避免连续上传时闭包里的 `photos` 过期。

### P1：照片大小限制过硬，缺少压缩/重试体验

现象：用户反馈“照片不能超过 5m”。当前客户端和服务端都硬限制 5MB。

代码证据：

- `src/components/profile/photo-uploader.tsx:21` 和 `src/app/api/profile/photos/route.ts:18` 均为 5MB。
- `src/components/profile/photo-uploader.tsx:78`、`src/app/api/profile/photos/route.ts:125` 直接报错。

建议修复：

- 客户端上传前压缩和转 WebP，目标 1600px 长边、质量 0.8 左右。
- 服务端保留安全上限，但可以提高到 8-10MB，再统一转码压缩。
- 错误文案给出可执行建议：“当前 7.2MB，建议自动压缩后重试/请换用原图压缩模式”。

### P1：匹配算法未实现地区期待条件

现象：用户希望“地理更细致加入匹配环节”，也提到同省符合人数不明显。项目文档和 Prisma 模型有地区匹配设计，但当前匹配核心函数没有使用。

代码证据：

- `prisma/schema.prisma` 的 `Preference` 包含 `locationScope`、`expectedProvinceCode`、`expectedCityCode`。
- `src/lib/validations/profile.ts` 的 preference schema 没有这些字段。
- `src/app/(auth)/signup/page.tsx`、`src/app/(auth)/complete-profile/page.tsx`、`src/app/(main)/profile/edit/page.tsx` 只收集用户自己的所在地，没有收集“期待所在地”。
- `src/lib/matching.ts` 的 `accepts` 只检查年龄、身高、体重、属性，没有地区判断。

建议修复：

- 在资料/偏好表单增加“期待所在地”：不限、同省、同城、指定省市。
- 补齐 `profileFormSchema` 和 `/api/profile/me` 的字段读写。
- 在 `MatchCandidate.preference` 增加 location 字段，并在 `accepts` 中实现城市/省份/不限逻辑。
- 匹配页同省筛选可以保留，但它应作为视图筛选，不能替代算法硬条件。

### P1：评分端 API 暴露了不必要的用户资料

项目文档要求评分组只能看到照片，不能看到被评分用户的身份资料。当前接口虽然页面主要展示照片，但 API 返回了出生日期、身高、体重、属性等资料，属于隐私面过宽。

代码证据：

- `src/app/api/scoring/tasks/route.ts:48-56` 查询 `ratedUser.profile` 的 `birthDate`、`heightCm`、`weightKg`、`attribute` 等字段。
- `src/app/api/scoring/tasks/route.ts:100-106` 又将这些字段写入响应。

建议修复：

- 评分任务接口只返回任务 ID、照片 URL、评分进度、必要状态，不返回资料字段。
- 如果管理端需要资料，应走管理端接口，按角色和审计记录访问。
- 增加接口测试，确保评分员角色拿不到 `profile.birthDate / heightCm / weightKg / attribute`。

### P1：移动端浏览器兼容问题

现象：

- “评分得换浏览器才能看到”。
- “资料填写前半部分选择题在夸克里不显示已选择”。

可能原因：

- 资料页大量使用自定义按钮状态、HSL CSS 变量和原生 range/select 混合控件。
- 若某些浏览器对 `hsl(var(--primary)/0.15)`、叠层 range、深色模式变量、focus/hover 状态支持异常，已选状态可能不可见。

建议修复：

- 建立移动端兼容测试矩阵：QQ 浏览器、夸克、微信内置浏览器、Safari iOS、Chrome Android。
- 已选状态不要只依赖背景色，增加勾选图标、边框粗细、文字状态。
- 对 `DualRangeSlider` 和 `.slider-input` 做触控命中区域测试。
- 若评分页依赖角色导航，确认移动端底部导航和评分入口在 SCORER/ADMIN 下稳定展示。

### P2：照片驳回与资料修改冷却策略过硬

现象：多位用户反馈 7 天冷却太长，尤其“照片被打回后还得等七天才能重新上传”和“修改信息间隔能调短到 3/5 天”。

代码证据：

- `src/lib/validations/profile.ts:17` 设置 `EDIT_COOLDOWN_DAYS = 7`。
- `src/app/api/profile/me/route.ts:190-197` 阻止 7 天内再次发布。
- `src/app/(main)/profile/edit/page.tsx:534` 前端提示发布后 7 天内不能再次修改发布。

建议处理：

- 产品上区分“资料发布冷却”和“照片审核驳回重传冷却”。
- 资料核心字段可保留 7 天，但照片被驳回应允许立即或短冷却后重传，避免审核失败造成进阶路径中断。
- 冷却天数改为可配置项，方便内测期调整为 3/5/7 天做观察。

## 4. 需新增的功能

### 4.1 匹配维度扩展

问卷中多次提到希望不只靠外在条件：BMI、地理细分、未来规划、爱好、思想、游戏兴趣、具体兴趣爱好、身材分。

建议拆成两类：

1. 可计算维度：BMI、同城/同省、资料完整度。
2. 用户填写维度：兴趣爱好、常玩游戏、未来规划、关系期待、价值观关键词。

落地建议：

- BMI 不一定需要新数据库字段，可由 `heightCm` 和 `weightKg` 计算展示，并可提供 BMI 范围偏好。
- 兴趣爱好先做轻量标签，多选 + 自定义补充，避免大文本难以匹配。
- 未来规划/价值观先做展示和搜索加权，不建议初期作为硬筛选，否则会让用户感觉“层层卡”。

### 4.2 单向匹配轻量资料卡

当前单向匹配只展示匹配项是否符合和原因，隐私保护较强，但用户觉得“资料过少”。

建议新增脱敏摘要：

- 昵称或匿名代称：可根据隐私策略决定。
- 年龄段而非精确年龄，如 18-22、23-26。
- 省份/城市可按当前可见规则展示。
- MBTI、兴趣标签、自我介绍前 60 字。
- “为什么不是双向”的解释保留，但放在折叠区域。

同时保留查看申请，用户想看完整资料时再发起互相解锁。

### 4.3 举报一键化

新增从匹配卡片、详情页、查看申请页进入举报时的闭环：

- 自动绑定目标用户，不让用户手填 QQ。
- 展示目标昵称/匿名代称，避免误报。
- 自动附带来源上下文：来自双向匹配、单向匹配、资料详情或查看申请。
- 保留证据上传，多图证据已经在举报页有较好基础。

### 4.4 照片上传增强

建议新增：

- 多图选择和批量上传。
- 上传前压缩。
- 上传进度和失败重试。
- 拖拽/长按排序。
- 上传后明确显示“已进入评分/还需提交资料后进入评分”。

### 4.5 内测反馈入口

这次问卷暴露出若干截图附件，但系统内缺少结构化反馈入口。建议在用户端增加“反馈问题”按钮：

- 自动带当前页面、浏览器、用户角色、屏幕尺寸。
- 可上传截图。
- 可选问题类型：注册、资料、照片、评分、匹配、举报、其他。

## 5. 需优化的交互体验

### 5.1 注册/验证码/密码流程

问题：

- 用户不知道为什么要 QQ 邮箱验证。
- QQ 邮箱验证码可能进垃圾箱。
- 登录页和注册页之间切换不够显眼，有用户把验证码输到密码栏。
- 注册页底部提示小字过小。

建议：

- 登录页“首次注册”按钮视觉提升，不要只像一个小链接。
- 注册页第一步标题改为“输入 QQ 邮箱验证码”，副文案加“验证码不是登录密码”。
- Bot 回复和注册页都提示“若收件箱没有，请检查垃圾箱/广告邮件”。
- 注册页顶部展示流程条：群内发送注册指令 -> 收 QQ 邮箱验证码 -> 设置登录密码 -> 完善资料。
- 验证码页增加“我还没有验证码/如何获取验证码”的明显入口。

### 5.2 滑杆输入

问题：用户多次提到身高、体重、年龄范围、评分滑杆在手机上不方便。

建议：

- 单值字段：保留滑杆，同时增加数字输入框和 `- / +` stepper。
- 范围字段：`DualRangeSlider` 下方增加两个数字输入框，避免双滑块重叠时难拖。
- 评分页：每项评分增加 `-0.1 / +0.1` 或 `-0.5 / +0.5` 按钮，并提供常用分段快捷按钮。

### 5.3 匹配页信息密度

问题：用户觉得系统“都在筛选，完全没聊，层层卡”。

建议：

- 匹配结果里增加“破冰信息”：兴趣标签、常玩游戏、未来规划摘要。
- 对单向匹配解释要更自然：少用“我符合他/他符合我”作为唯一理解入口，可改为“对方符合你的期待 / 你符合对方期待”。
- 同省筛选的数量要直接展示，例如“同省 3 人 / 全部 12 人”。

### 5.4 冷却提示

问题：用户只感受到“不能改/不能传”，不清楚为什么、还能做什么。

建议：

- 冷却提示里明确“可先保存草稿，冷却结束后发布”。
- 冷却倒计时用具体日期：如“2026-07-04 之后可再次发布”。
- 照片驳回场景单独提示“你可以修改照片并重新提交”的条件，不和资料冷却混在一起。

## 6. 推荐优先级路线

### 第一批：立即修复

1. 修复举报传参：内部 ID 和 QQ 字段拆开。
2. 照片上传支持多选，至少先做到逐张自动上传。
3. 增加上传前压缩或提高服务端上限并转码。
4. 移除评分任务 API 中不必要的资料字段。
5. 注册页/登录页文案和入口显著性调整。

### 第二批：内测体验增强

1. 身高、体重、年龄范围、评分滑杆增加数字输入和加减按钮。
2. 夸克/QQ 浏览器兼容修复。
3. 冷却文案和倒计时优化。
4. 单向匹配页增加轻量资料摘要。

### 第三批：产品能力扩展

1. 实现期待所在地字段和地区匹配算法。
2. 增加 BMI 展示和 BMI 偏好。
3. 增加兴趣、游戏、未来规划、价值观等资料字段。
4. 让匹配排序引入资料完整度、兴趣重合度、同省/同城加权。
